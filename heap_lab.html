Why an RTOS needs a heap

  • Tasks, queues, semaphores, timers created with xTaskCreate etc. need RAM for their TCB + stack. That RAM has to come from somewhere — a heap is just "somewhere with bookkeeping."
  • Unlike desktop OSes, there's no MMU/paging safety net on most MCUs. A bad allocator can silently corrupt the whole system — determinism and overhead matter more than on Linux.
  • FreeRTOS ships five interchangeable heap files (heap_1..5) — you pick one by adding it to your build. Same API (pvPortMalloc/vPortFree), wildly different internals.

Fragmentation, in one picture

Internal fragmentation: wasted space inside a block you already own (alignment padding, or an allocator that hands you more than you asked for).

External fragmentation: wasted space between blocks — plenty of total free memory, but no single hole big enough. This is the one that kills long-running embedded systems.

The vocabulary you'll use in every lab below

  • Metadata / header: a small struct FreeRTOS hides just before every block (size, and for free blocks a next pointer). It's overhead you pay per allocation.
  • Free list: a linked list threading together every free block. Allocation = search this list. Free = insert into this list.
  • Splitting: found a free block bigger than you need? Chop it — carve out exactly what's needed, leave the remainder as a smaller free block.
  • Coalescing: when a block is freed, merge it with a physically adjacent free neighbor so two small holes become one big one. This is the single biggest lever against external fragmentation.
  • Deterministic: allocation takes bounded, predictable time regardless of heap state — critical for hard real-time. Any list scan (best-fit, first-fit) is technically O(n) and not strictly deterministic; a bump allocator (heap_1) is O(1) and is.

Design philosophy

The dumbest possible allocator: a bump pointer. Every pvPortMalloc carves the next slice off the top of a static array and moves the pointer forward. There is no vPortFree. Calling it compiles, but does nothing.

  • Zero free list, zero metadata search → O(1), fully deterministic.
  • Zero fragmentation risk, because nothing is ever freed or reused.
  • Smallest code size of all five heaps.

When to use it for real

Systems that create all tasks/queues/semaphores once at startup and never delete them — the vast majority of small embedded products. If you never call vTaskDelete, heap_1 is often the correct choice, not a limitation.

Interview traps

  • "Does heap_1 leak memory?" — no, leaking implies you lost a valid pointer. Here memory is intentionally permanent.
  • "Is heap_1 thread-safe?" — the bump pointer update should still happen inside a critical section against concurrent task/ISR allocation.

Design philosophy

Adds a real free list on top of heap_1's idea: vPortFree now works. Allocation uses best-fit — it scans the entire free list and picks the smallest block that's still big enough, to minimize leftover slack.

The catch: freed blocks are never merged with physically adjacent free neighbors. Two 32B holes sitting right next to each other stay two 32B holes forever — even though together they could satisfy a 60B request.

Advantages / disadvantages

  • + Supports free(), better slack control than a naive first-fit.
  • − O(n) search every allocation — not great for hard real-time.
  • − No coalescing → heavy external fragmentation on long-running alloc/free churn.
  • FreeRTOS docs explicitly say: superseded by heap_4, don't use for new designs.

When to use it for real

Rare today. Maybe: allocations are always near-uniform size (so best-fit slack is naturally tiny and adjacency never matters), and you specifically don't want the extra coalescing code/time cost of heap_4. In practice: just use heap_4.

Design philosophy

heap_3 isn't really a FreeRTOS allocator at all — it's a thin, thread-safe wrapper around the standard C library's malloc()/free(). FreeRTOS just suspends the scheduler (or wraps a critical section) around the call so a context switch can't interrupt libc's internal bookkeeping.

Advantages / disadvantages

  • + Reuses a mature, battle-tested allocator — often first-fit-with-coalescing like heap_4, but the exact algorithm is whatever your toolchain ships (newlib, picolibc, dlmalloc…).
  • + Heap size isn't fixed by configTOTAL_HEAP_SIZE — it's whatever the linker/compiler gives the C runtime.
  • − Not deterministic and typically slower — you inherit libc's worst case, not FreeRTOS's.
  • − Bigger code size (you link the whole libc allocator).

When to use it for real

Porting desktop/POSIX-ish code to an RTOS quickly, or a platform (Linux-based RTOS shim, simulator build) where libc malloc is already required elsewhere.

This lab models heap_3's behavior as first-fit + coalescing (a common real allocator shape) purely so you can compare its fragmentation story to heap_4 — treat the exact search order as illustrative, since real libc internals vary.

Log

Design philosophy

FreeRTOS's own general-purpose allocator, and the default recommendation. Free blocks are kept in address order. Allocation is first-fit: walk the list, take the first block big enough. On free(), the block checks its immediate neighbors in the list — if either is free and physically adjacent, merge.

Advantages / disadvantages

  • + Coalescing keeps holes big, dramatically reducing external fragmentation vs heap_2.
  • + First-fit is cheaper per-search than best-fit's exhaustive scan.
  • + Exposes xPortGetFreeHeapSize() / xPortGetMinimumEverFreeHeapSize() for live diagnostics (only heap_4/5).
  • − Still O(n) worst case → not hard-real-time deterministic.
  • − First-fit can still eat into a big block for a tiny request (compare with the demo below).

When to use it for real

Default choice for anything that creates and deletes tasks/queues dynamically at runtime. If in doubt in an interview: heap_4.

vs heap_2

Same "has a free list" idea, but heap_4 trades best-fit's exhaustive search for first-fit's speed, and spends that budget on coalescing instead — which matters far more for long-run fragmentation than picking the mathematically tightest hole.
